端系统运行传输层协议

发送方：将应用消息分成一个或多个的Segment,并向下传给网络层
接收方：将接收到的Segment组装成消息，并向上交给应用层
应用层可以提供多种协议：Internet上的TCP和UDP

网络层 vs. 传输层

网络层：提供主机之间的逻辑通信机制
传输层：提供应用进程之间的逻辑通信机制
1.依赖于网络层
2.位于网络层之上
3.对网络层服务进行（可能的）增强

TCP

拥塞控制
流量控制
连接建立

UDP

best-effort

多路复用和多路分用

原因

如果某层的一个协议对应直接上层的多个协议/实体，则需要复用/分用

举例

简单来说，传进来是多路复用，传出去是多路分用。
多路复用发生在传输层，多路分用发生在网络层。

如何工作

1.主机接收到IP数据报
2.主机接收到segment之后，传输层协议提取IP地址和端口号信息，将segment导向相应的secket

TCP的socket用四元组标识：
源ip地址
源端口号
目标ip地址
目标端口号

接收端利用所有的四个值将segment导向合适的socket

可靠数据传输原理

rdt1.0

1.底层信道完全可靠
不会发生错误(bit error)
不会丢弃分组
2.发送方和接收方的FSM独立

rdt2.0

产生位错误的信道
利用校验和来检测位错误

如何从错误机制中恢复？

确认机制(Acknowledgements, ACK): 接收方显式地告知发送方分组已正确接收
NAK, 接收方显式地告知发送方分组有错误
发送方收到NAK后重传

基于这种重传机制的rdt协议成为ARQ(Automatic Repeat reQuest)协议

Rdt2.0中引入的新机制

1.差错检测
2.接收方反馈控制信息：ACK/NAK
3.重传

设计

rdt2.1

rdt2.0的缺陷

如果ACK/NAK消息发生错误/被破坏(corrupted)会怎么样？

为ACK/NAK增加校验和，检错并纠错
发送方收到被破坏ACK/NAK时不知道接收方发生了什么，添加额外的控制消息
如果ACK/NAK坏掉，发送方重传
不能简单重传：产生重复分组

如何解决重复分组问题？

序列号：发送方给每个分组增加序列号
接收方丢弃重复分组

发送方

为每个分组增加了序列号
两个序列号（0，1）就够用，为什么？——停等协议
需校验ACK/NAK消息是否发生错误
状态数量翻倍——状态必须记住当前分组的序列号

接收方

需判断分组是否重复——当前所处状态提供了期望收到分组的序列号
注意：接收方无法知道ACK/NAK是否被发送方正确接收到

rdt2.2

只需要ACK实现

如何实现

1.接收方通过ACK告知最后一个被正确接收的分组
2.在ACK消息中显示地加入被确认分组的序列号

发送方收到重复ACK后，采取与收到NAK消息相同的动作——重传当前分组

rdt3.0

如果信道既可能发生错误，也可能丢失分组，怎么办？

方法

控制发送方等待合理的时间
如果没有收到ACK，重传
如果分组只是延迟而不是丢了
1.重传能够重复，序列号机制能够处理
2.接收方需在ACK中显示告知所确认的分组
需要定时器

Go-Back-N

GBK的发送方必须响应三种类型的事件：

上层的调用
收到一个ack

在GBK协议中，对序号为n的分组的确认采取累计确认(cumu ackonwledgment)的方式，表明接收方已经正确接收到序号为n的以前且包括n在内的所有分组。

超时事件

GBK接收方动作

如果一个序号为n的分组被正确的接收到，并且按序，则接收方为分组n发送一个ACK，并将该分组的数据部分交付到上层。在所有其他情况下，接收方丢弃该分组，并为最近按序接收的分组重新发送ACK。注意到因为一次交付给上层一个分组，如果分组k已接收并交付，则所有序号比k小的分组也已经交付。

特点

点对点
一个发送方，一个接收方
可靠的、按序的字节流
流水线机制
TCP拥塞控制和流量控制机制设置窗口尺寸

3.TCP的连接包括：两台主机上的缓存、连接状态变量、socket等

TCP段结构

序列号：
指的是segment中第一个字节的编号，而不是segment的编号
建立TCP连接时，双方随机选择序列号
ACKs：
希望接收到的下一个字节的序列号
累计确认：
该序列号之前的所有的字节都已经被正确接收到了
接收方如何处理乱序到达的segment？
TCP规范中没有约定，实现者做出决策

TCP可靠数据传输

如何设置定时器超时的时间

大于RTT(但RTT是变化的)
过短(不必要的重传)
过长(对段丢失时间反应慢)

如何估计RTT

SampleRTT：测量段发出去到收到ACK的时间
SampleRTT的变化：测量多个SampleRTT，求平均值，形成RTT的估计值
计算公式：
EstimatedRTT = (1-α) * EstimatedRTT + α * SampleRTT

定时器超时时间的设置：
EstimatedRTT + '安全边界'
EstimatedRTT变化较大 -> 较大的边界

测量RTT的变化值：SampleRTT与EstimatedRTT的差值
DevRTT = (1-β)*DevRTT + β*|SampleRTT - EstimatedRTT| (typically, β = 0.25)

定时器超时时间设置：
TimemoutInterval = EstimatedRTT + 4 * DevRTT

TCP流量控制

控制发送方的发送速度不要过快，本质上是一种速度匹配机制

即使是RevWindow = 0，发送方也可以发送一个很小的段

TCP连接管理

三次握手

第一阶段

客户主机向服务器发送一个SYN报文段（不携带任何数据，SYN标志位要置1，传递选择的初始的序列号）

第二阶段

服务器收到SYN报文段，如果要建立连接答复SYNACK报文段，服务器会分配缓存，选择自己初始的序列号，并告知服务器

第三阶段

服务器收到SYNACK报文段，服务器会答复一个ACK报文段，SYN置0，这个报文段可以携带数据

关闭连接

第一步

客户机向服务器发送TCP FIN控制报文段

第二步

服务器收到FIN，回复ACK，关闭连接，发送FIN

第三步

客户机收到FIN，回复ACK
进入“等待”，如果收到FIN会重新发送ACK

第四步

服务器收到ACK，连接关闭

拥塞控制

成因

太多的数据发送方发送了太多数据

拥塞代价：

分组时延太大
当分组被drop时，任何处于该分组的“上游”传输能力全都被浪费掉

拥塞控制的方法

端到端的拥塞控制
1.网络层不需要显式的提供支持
2.端系统通过观察loss，delay等网络行为判断是否发送拥塞
TCP采用这种方法
网络辅助的拥塞控制
1.路由器向发送方显式地反馈网络拥塞信息
2.简单的拥塞指示(1bit):SNA, DEbit, TCP/IP ECN, ATM
3.只是发送方应该采取何种速率

案例 ATM ABR拥塞控制

ABR： available bit rate
“弹性服务”
如果发送方路径underloaded，使用可用带宽
如果发送方路径拥塞，将发送速率降低到最低保障速率
RM(resource management) cells： 资源管理cell
发送方发送
交换机设置RM cell位(网络辅助)
NI bit rate：不许增长
CI rate：拥塞指示
RM cell： 由接收方反馈给发送方

TCP拥塞控制

如何来控制发送速率

LastByteSent - LastByteAcked <= Congwin
rate ≈ Congwin/RTT (Bytes/sec)
CongWin
动态调整以改变发送速率
反应所感知到的网络拥塞

如何感知网络拥塞

LOSS事件 = timeout或3个重复的ACK
发生loss时间，发送方降低速率

如何合理的调整发送速率
加性增—乘性减：AIMD

原理：逐渐增加发送速率，谨慎探测可用带宽，直到发生loss
方法：AIMD
Additive Increase:每个RTT将CongWin增大一个MSS——拥塞避免
Multiplicative Decrease:发生loss后将CongWin减半

慢启动：SS

TCP连接建立时，CongWin=1
例：MSS = 500 byte, RTT = 200 msec. 初始速率 = 20k bps
可用带宽可能远远高于初始速率，所以我们希望可以进行指数增长
原理：当连接开始时，指数性增长

何时应该将指数型增长切换为线性增长

当CongWin达到Loss事件当前值1/2时。
实现方法
变量：Threshold
Loss事件发生时，Threshold被设为Loss事件前CongWin值的1/2.

LOSS事件的处理

三个重复的ACK
CongWin切到一半，然后线性增长
Timeout事件
CongWin直接设为1个MSS，然后指数增长，达到threshold后再线性增长
原因
3个重复的ACK表示网络还能够传输一些segments，timeout事件表明拥塞更为严重

静态路由 vs. 动态路由

静态路由

手工配置
路由更新慢
优先级高

动态路由

路由更新快：
1.定期更新
2.及时响应链路费用或网络拓扑变化

全局信息路由算法 vs. 分散信息路由算法

全局信息

所有路由器掌握完整的网络拓扑和链路费用信息，eg：链路状态(LS)路由算法

分散信息

路由器只掌握物理相连的邻居以及链路费用
邻居间信息交换、运算的迭代过程，eg：距离向量(DV)路由算法

链路状态路由算法

Dijkstra算法

所有结点(路由器掌握网络拓扑和链路费用)
·通过链路状态广播
·所有结点拥有相同的信息
计算从一个结点(“源”)到达所有其他点的最短路径
·获得该结点的转发表
迭代k次后，得到到达k个目的结点的最短路径。
c(x,y)结点x到结点y链路费用；如果x和y不直接相连，则=∞
D(v):从源到目的v的当前路径费用值
p(v):沿从源到v的当前路径，v的前序结点
N‘:已经找到最小费用路径的结点集合

算法描述

时间复杂度：n(n+1)/2
更好的实现nlogn
存在震荡现象

距离向量(Distance Vector)算法

Dx(y) = 从结点x到结点y的最小费用估计
·x维护向量(DV):
结点x：
·已知到达每个邻居的费用c(x,y)
·维护其所有邻居的距离向量：

核心思想

每个结点不定时地将其自身的DV估计发送其邻居
当x接收到邻居的新的DV时，即依据B-F更新其自身的距离向量估计：

Dx(y)将最终收敛于实际最小费用

特点

1.异步迭代
(1)引发每次局部迭代的因素
1)局部链路费用改变
2)来自邻居的DV更新
2.分布式
(1)每个结点只当DV变化时才通告给邻居
1)邻居在必要时(其DV更新后发生改变)再通告它们的邻居

本协议主要用于支持使用客户端远程管理在服务器上的电子邮件。
具体过程：特许—>用户代理发送(以明文的形式)用户名和口令以鉴别用户时间处理—>用户代理取回报文，同时可以对报文做删除，标记和取消报文删除标记以及获取邮件统计信息

TCP Transmission Control Protocol 传输控制协议

是一种面向连接的、可靠的、基于字节流的传输层通信协议

RDT Reliable Data Transfer Protocol 可靠数据传输协议

就是采用一系列技术来保障信息在发送方和接收方准确、精确的传输。

Go Back N 回退N步/滑动窗口协议

允许发送方发送多个分组，而不需要等待确认，但它也受限于在流水线中未确认的分组数不能超过某个最大允许数N。

SR Selective Repeat 选择重传

通过让发送方仅重传那些它怀疑在接收方出错的分组而避免了不必要的重传。

ICMP Internet control message protocol Internet控制报文协议

用于在IP主机，路由器之间传递控制消息的协议。典型用途是差错报告。

Rip Routing Information protocol 路由信息协议

是一种动态路由选择协议，用于自治系统（AS）内的路由信息的传递

Ospf Open shortest Path First 开放式最短路径优先

是一个内部网关协议，用于在单一自治系统内决定路由。是一种链路状态协议，使用洪泛链路状态信息和Dijkstra最低开小路径算法。

CSMA/CD Carrier Sense Multiple Access/collision Detection 带冲突检测的载波监听多路访问技术

局域网特点有哪些？描述CMSA/CD的基本思想

局域网的特点：
（1）地理范围比较小。
（2）数据传输速率高。
（3）误码率低。
（4）局域网一般为一个单位或部门所独有。
C S M A / C D协议的工作原理是：某站点想要发送数据，必须首先侦听信道。如果信道空闲，立即发送数据并进行冲突检测；如果信道忙，继续侦听信道，直到信道变为空闲，才继续发送数据并进行冲突检测。如果站点在发送数据过程中检测到冲突，它将立即停止发送数据并等待一个随机长的时间，重复上述过程。

简述集线器和交换机的区别

集线器工作在物理层，只用作信号的整形和放大，相当于电子总线。交换机工作在链路层，可以在不同的端口之间按不同的MAC地址同时快速交换数据，起到多口网桥的作用。
